# Agent O — Project Goal

## The problem

The current AI software solution has two agents: Agent Y (Knowledge Intelligence — a chatbot that answers from company knowledge bases) and Agent I (Intelligence Automation — browser automation for proprietary internal software). Each agent is a standalone system built for a specific capability.

This architecture is not scalable. Every time a new capability is needed, a new agent must be built from scratch. The system acts as a rigid router — requests go to Agent Y or Agent I and nothing else. Adding capability #3 means building agent #3. Adding capability #10 means building agent #10.

## The solution

Build Agent O — an orchestration layer agent that sits between the messaging gateway and the existing agents/tools. Agent O receives user requests, reasons about what is needed, and either executes tasks directly using skills and tools, or delegates to existing agents as sub-agents, or combines multiple agents and tools in a single workflow.

New capabilities become new skills, not new agents.

## What Agent O is built on

Agent O is built on the Hermes Agent architecture (Nous Research, open source, MIT license, Python). Hermes is a persistent, self-improving AI agent framework with five core pillars: Memory, Skills System, Soul, Crons, and Self-Learning Loops. Agent O retains these pillars and extends them for enterprise use.

## Core capabilities

### Skills and tools system

In Agent O, a **tool** is a single capability — each integration (Agent Y, IBM Maximo API, Oracle Procurement API, Claude Cowork job queue, etc.) is registered as a tool that Agent O can call.

A **skill** is a higher-level workflow that tells Agent O which tools to call and in what order. For example, a skill might say: "first call Agent Y for maintenance history, then call Maximo API for inventory, then call Oracle API for procurement." Tools are the building blocks. Skills are the workflows.

Skills are stored as SKILL.md files — structured markdown documents with sections for when to use the skill, the procedure, known pitfalls, and verification steps. Skills can be pre-authored by the team or auto-generated by Agent O from experience.

This is what makes Agent O scalable: adding a new capability means writing a new skill (or having Agent O learn one), not building a new agent.

### Self-learning

After completing a complex task (one involving multiple tool calls), Agent O evaluates what it did, extracts the workflow as a reusable skill, and stores it. The next time a similar task arises, Agent O already has the skill and can execute the entire workflow proactively rather than being guided step by step.

The skill is not programmed by a developer. It is written by Agent O itself from its own experience. The more Agent O is used, the more skills it accumulates, and the better it gets.

### Sub-agent spawning

Agent O can spawn isolated sub-agents for specific sub-tasks. Each sub-agent gets a scoped context (only the information relevant to its sub-task), executes independently, and returns results to Agent O. Sub-agents can run in parallel.

This enables scale (process 200 clients in parallel instead of sequentially) and data isolation (each sub-agent only sees its assigned client's data — no cross-contamination).

### Orchestration loop

Agent O uses a ReAct loop (Reason → Act → Observe) to process requests. It reasons about what is needed, selects an action (call a tool, retrieve a skill, delegate to a sub-agent, or respond to the user), executes the action, observes the result, and loops until the task is complete.

### Memory (tiered)

Agent O retains context across sessions using a tiered memory system inherited from Hermes:

- **Tier 1 (always loaded):** user context and working notes, loaded into every request
- **Tier 2 (on-demand):** full-text search across past sessions
- **Tier 3 (progressive disclosure):** skills library — summaries loaded first, full detail only if needed
- **Tier 4 (background):** durable factual memory — stores observations, facts, and entity relationships across sessions (self-hosted Hindsight instance in the company AWS VPC). Note: this provides factual knowledge recall and entity graph memory, not behavioural user modelling.

In enterprise, memory must be isolated per tenant and per user.

### Soul

A persistent personality and behavioral configuration that prevents tone drift over time. In enterprise, this is configurable per tenant — each client organisation can have its own agent persona.

### Crons

Scheduled task execution without human initiation. Agent O can run workflows on a timer (e.g., "every Monday at 7am, generate the executive briefing"). Cron jobs run through the same full agent pipeline with access to memory and skills.

## Enterprise requirements

Agent O must support four capabilities that personal Hermes never needed. These are what make Agent O enterprise-ready:

### 1. Permission-scoped execution

Every skill call, agent delegation, and data access is parameterised with the caller's permission scope. The same skill produces different results depending on who invoked it. This covers cross-departmental coordination, data governance, delegation chains, and multi-user shared context.

### 2. Human-in-the-loop with pause/resume

The orchestrator can prepare a task, pause for human approval (potentially for hours or days), and resume when approved. This covers approval workflows — procurement, hiring, contract reviews, deployments, content publishing. Agent O prepares everything but does not execute consequential actions without human sign-off.

### 3. Audit trail

Every decision Agent O makes produces an immutable, traceable log entry — what data was accessed, which skill was used, what reasoning was applied, who authorised the action. Required for regulated industries (HIPAA, SOC 2, financial regulations).

### 4. Non-human triggers

Agent O accepts tasks from APIs, webhooks, business events, and inter-agent escalation — not just user chat messages. This enables system-triggered workflows (monitoring detects anomaly → Agent O investigates), event-driven chains (new customer signed → Agent O runs onboarding), and inter-agent escalation (Agent Y can't answer → escalates to Agent O).

## The agent ecosystem

Agent O does not work alone. It orchestrates across a growing ecosystem of agents and tools:

| Agent / System | Responsibility | Data domain |
|---|---|---|
| **Agent O** | Orchestration and intelligence synthesis. Receives requests, retrieves skills, calls tools, spawns sub-agents, learns from experience. | Orchestrates across all domains. |
| **Agent Y** | Knowledge Intelligence. Answers from company knowledge bases. Case analysis and recommendation. Priority integration. | Internal: company KBs, proprietary data, SOPs, policies, historical records. |
| **Agent I** (future) | Intelligence Automation. Browser automation for proprietary internal software. Navigates internal web-based tools. Deferred — harder to integrate. | Internal live systems: operational dashboards, monitoring tools, ERP, CRM. |
| **Claude Cowork** | Specialised research agent. Performs deep web research via a job queue. Agent O writes research briefs; Cowork returns structured results. | External: web, public data, competitor information, industry news. |
| **IBM Maximo API** | Enterprise asset management. Inventory and spare parts stock levels. | Internal: asset and inventory data. |
| **Oracle Procurement API** | Procurement system. Purchase requisitions and order submission. | Internal: procurement and supplier data. |

Agent Y is the priority integration. Agent I integration is deferred (harder to integrate). External research is delegated to Claude Cowork via a job queue, not handled by Agent O directly. Maximo and Oracle are integrated via API calls.

## Use cases

Three use cases demonstrate what Agent O should be able to do. Each shows a different Agent O capability:

### Use Case 1: Incident Response Coordination (Operational)

**Demonstrates:** skills and tools system, multi-tool orchestration, human-in-the-loop, self-learning.

A crane experiences hydraulic failure. An engineer reports the problem to Agent O. Agent O calls Agent Y for maintenance history and SOPs (tool call), synthesises an action plan and parts list, checks IBM Maximo for inventory (tool call), prepares a procurement request for out-of-stock parts, pauses for engineer approval (human-in-the-loop), and submits via Oracle Procurement (tool call). After completion, Agent O learns the workflow and writes a new skill. Next time a similar failure occurs, a different engineer describes symptoms and Agent O runs the entire workflow proactively — one prompt, one approval, done.

### Use Case 2: Market Intelligence Report (Commercial)

**Demonstrates:** three-agent architecture, job queue delegation, preliminary research for human analysis, self-learning.

A commercial team needs a competitor analysis. Agent O writes a structured research brief to a job queue. Claude Cowork picks up the task and performs deep preliminary research (web browsing, public filings, industry news). Agent O receives the results and calls Agent Y for internal data (own performance, client overlap, pricing). Agent O synthesises both into a preliminary report. The analyst team performs secondary analysis — reviewing, interpreting, forming strategic views, adding recommendations. Agent O augments the analyst team, not replaces them. After completion, Agent O writes a skill for this competitor type.

### Use Case 3: Regulatory Update Propagation (Operational / Compliance)

**Demonstrates:** sub-agent spawning, parallel processing, data isolation, enterprise governance, self-learning per jurisdiction.

A country updates its customs regulations. A compliance officer hands the gazette notice to Agent O. Agent O calls Agent Y to interpret the change and identify affected clients. Agent O spawns one sub-agent per affected client — each sub-agent gets only its assigned client's data (data isolation), prepares configuration changes, and drafts a client notification independently and in parallel. Agent O aggregates all results into a single approval package. The compliance officer reviews and approves. Agent O applies changes and sends notifications. After completion, Agent O writes a skill for this country's regulatory pattern. The skill library grows per jurisdiction.

## Big picture goal

The current software solution should have the same architecture, features, and benefits as Hermes, adapted for enterprise multi-tenant use. Agent O is the orchestration brain that makes this possible — receiving requests, reasoning about what is needed, leveraging a growing library of skills, calling the right tools and agents, learning from experience, and getting better over time. New capabilities are skills, not new agents.
